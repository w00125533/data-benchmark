# StarRocks HDFS Parquet Load, Smoke Validation, and Spark Native SQL Design

Date: 2026-06-06
Status: Draft for review

## Context

The KPI benchmark currently generates Parquet data and loads or exposes it through several query routes. The recent 1B benchmark proved the HDFS dataset exists and Spark Iceberg can load it, but StarRocks internal can still appear empty because the KPI internal path depends on a separate CSV export and Stream Load path. For HDFS outputs, that internal load path is skipped to avoid generating another large local CSV artifact.

The desired direction is to make HDFS Parquet the single generated dataset and have each engine either import from it or register/query it directly. Smoke should fail when an engine reports success but contains no data.

## Goals

- Generate KPI data once as Parquet on HDFS.
- Load StarRocks internal from the generated HDFS Parquet dataset, not from CSV.
- Keep StarRocks external Iceberg and Hive/Spark routes aligned to the same dataset.
- Add smoke checks that prove data exists in every configured storage/query shape.
- Add a Spark native SQL route that queries the generated Parquet directly, so it can be compared against Spark Iceberg.
- Remove the KPI CSV export and KPI Stream Load dependency.

## Non-Goals

- Migrating TPC-H CSV loading in this change.
- Reworking shared infrastructure service ownership.
- Adding a new frontend report design.
- Changing KPI generator distribution beyond query compatibility fixes already made.

## Proposed Data Flow

The KPI compose benchmark should use this flow:

1. Spark generates Parquet to the configured dataset output path.
2. The runner records dataset rows and bytes from the generator result.
3. Spark native SQL registers a direct Parquet view or external table over the generated path.
4. Spark Iceberg creates the run-specific Iceberg table and inserts from the generated Parquet.
5. Hive creates or repairs an external Parquet table over the generated HDFS path.
6. StarRocks external refreshes the Iceberg external catalog/table.
7. StarRocks internal imports the generated HDFS Parquet files with Broker Load.
8. The runner validates each loaded/registered shape with `COUNT(*)`.
9. Query benchmark phases run only after the relevant route passed readiness and load validation.

## Table Shapes

The KPI schema table shape map should include:

- `spark_native_parquet`: Spark SQL direct Parquet route.
- `spark_iceberg`: Spark Iceberg table.
- `hive_hdfs_parquet`: Hive external Parquet table.
- `starrocks_external_iceberg`: StarRocks external Iceberg catalog table.
- `starrocks_internal`: StarRocks internal OLAP table loaded from HDFS Parquet.

The `BenchmarkRoute` enum should add `SPARK_NATIVE_PARQUET`. Existing route restart/readiness behavior can reuse the Spark service.

## StarRocks Internal Load

KPI StarRocks internal should move from CSV Stream Load to Broker Load from HDFS Parquet.

The client should:

1. Create the `sr_internal` database and `cell_kpi_1min` table.
2. Clear existing table contents before a run to prevent duplicate counts.
3. Submit a run-scoped load label.
4. Use `DATA INFILE("hdfs://hdfs-namenode:8020/.../*.parquet")`.
5. Specify `FORMAT AS "parquet"` and the KPI column list.
6. Use simple HDFS authentication properties.
7. Poll `information_schema.loads` until the label is `FINISHED` or `CANCELLED`.
8. Return a failed load result on cancellation, timeout, or missing load state.
9. Record loaded rows from `SINK_ROWS` or a follow-up `COUNT(*)`.

Broker Load is appropriate because StarRocks supports HDFS Parquet loading and runs the job in the background. This avoids keeping a local client session open for a long import and removes the second CSV copy.

## Data Validation

Add an explicit load validation stage after each route is prepared.

Validation should execute:

```sql
SELECT COUNT(*) FROM <table_shape>
```

The expected value is `dataset.rows()`.

Validation result rules:

- If count equals expected rows, record a successful check result.
- If count differs, record a failed check result with `expected=<n>, actual=<m>`.
- If the query fails, record a failed check result with the underlying error.
- If a route fails validation, its benchmark queries should be skipped or marked failed for that route. The run should be `DEGRADED` or `FAILED` according to the existing report status rules.

Suggested stages:

- `SPARK_NATIVE_PARQUET_VALIDATE`
- `SPARK_ICEBERG_VALIDATE`
- `HIVE_HDFS_PARQUET_VALIDATE`
- `STARROCKS_EXTERNAL_VALIDATE`
- `STARROCKS_INTERNAL_VALIDATE`

The current StarRocks external/internal load stages may still report zero bytes because refresh/import metadata is not equivalent to generated bytes. The validation stage is the authoritative data presence check.

## Spark Native SQL Route

Add a Spark client path for native Parquet:

- Register a temporary view or external Spark SQL table over the generated HDFS path.
- Use the same KPI query catalog templates by replacing `{table}` with the native Parquet view/table.
- Reuse the existing Spark SQL command builder and CLI row parser.
- Include the route in cold/warm/hot phases, with restart/readiness behavior equivalent to Spark Iceberg.

Preferred implementation is a persistent Spark SQL table or view created before route queries, because each `spark-sql -e` invocation may have a separate session. A table in a run-scoped database avoids relying on temporary view lifetime.

## CSV Removal Scope

For KPI compose runs:

- Remove `StarRocksCsvExporter` usage from `ComposeBenchmarkRunner`.
- Remove `starrocks_csv` compose prepare results.
- Remove KPI use of `StarRocksStreamLoadClient`.
- Remove `STARROCKS_STREAM_LOAD_URL` from benchmark-runner compose environment if it is no longer used by KPI paths.

TPC-H CSV exporter and TPC-H Stream Load can remain for now. They are outside this spec and should be migrated separately if needed.

## Error Handling

Broker Load errors must preserve enough detail to debug:

- submitted label
- HDFS path
- final StarRocks load state
- `ERROR_MSG` from `information_schema.loads`
- expected and actual row counts when validation fails

Timeouts should be configurable and long enough for 1B data. A default of several hours is acceptable for Broker Load, while smoke can complete quickly because its dataset is small.

## Testing Plan

Unit tests:

- StarRocks Broker Load SQL includes HDFS Parquet path, `FORMAT AS "parquet"`, and KPI column mapping.
- Broker Load polling succeeds on `FINISHED`.
- Broker Load polling fails on `CANCELLED` and includes `ERROR_MSG`.
- Broker Load polling fails on timeout or missing label.
- Compose runner validates dataset rows for Spark native, Spark Iceberg, Hive, StarRocks external, and StarRocks internal.
- Compose runner marks a route failed when validation count mismatches.
- Compose runner no longer calls KPI CSV export.
- Query catalog renders all KPI queries for `spark_native_parquet`.

Command verification:

- `mvn test`
- `docker compose -f docker-compose.yml config`
- Compose smoke run with all five table shapes enabled.
- For a 1B run, verify the report dataset rows and validation rows are `1000000000` for all enabled shapes, including `starrocks_internal`.

## Acceptance Criteria

- KPI smoke fails when any generated dataset, Spark native, Spark Iceberg, Hive, StarRocks external, or StarRocks internal count is zero or mismatched.
- StarRocks internal is populated from HDFS Parquet without generating KPI CSV files.
- StarRocks internal and external both have validation records in the report.
- Spark native Parquet appears as a distinct benchmark route and can be compared against Spark Iceberg.
- Existing Spark Iceberg, Hive, StarRocks external, and report behavior continue to pass tests.

## Implementation Decisions

- Broker Load timeout defaults to 6 hours. The value can be overridden by configuration or environment later, but the first implementation should use a fixed default to keep behavior predictable.
- Validation failure should mark only the affected route's benchmark queries as failed or skipped and should make the run `DEGRADED`. Dataset generation failure remains a run-level failure because no route can be trusted without a dataset.
